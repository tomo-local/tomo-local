## 職務要約

ERP システムの導入・カスタマイズ開発から採用プラットフォームのフルスタック開発まで、約 5 年にわたり幅広い領域で経験を積んできたエンジニアです。現在は、採用プラットフォーム企業にて、フロントエンド開発を中心にバックエンド開発・外部サービス連携・パフォーマンス改善まで横断的に担当しています。

技術面では、TypeScript/React によるフロントエンド開発と Go によるバックエンド開発を軸に、Kubernetes・Terraform・Google Cloud を活用したインフラ構築まで一貫して対応できます。Monaco Editor を活用したコードレビュー機能や Google Calendar API との連携による面接自動化機能など、ユーザー体験を起点とした機能設計・実装を得意としています。また、EXPLAIN ANALYZE を用いたクエリ最適化（20.6 倍高速化）やマイクロサービスの統合など、プロダクト全体の技術的負債の解消にも主体的に取り組んでいます。

業務外では、React Tokyo コミュニティの運営メンバーとして Discord Bot の開発・改善をリードし、スレッド通知機能の実装やコードベースのリファクタリング・自動テスト基盤を構築しました。業務・コミュニティ双方を通じて、チーム全体のコード品質と開発効率の向上に継続的に取り組んでいます。

##### プロジェクト詳細

**タグ削除機能の追加**

タグの追加のみが可能で削除できない制約があり、不要なタグが蓄積して選考管理の効率が低下していました。削除機能を実装してユーザーが柔軟にタグ管理できるようにし、選考管理の利便性を向上させました。

**インタビューのレポート共有機能の実装**

面接結果のレポートを外部の採用担当者と共有する手段がなく、情報共有が非効率でした。共有内容の設定・有効期限の設定・共有用 URL の生成機能を実装し、セキュアかつ柔軟な情報共有を実現しました。

**Google Meet・Zoom 連携による AI 面接評価機能の実現**

AI によるライブコーディングセッションの評価機能がアプリ内のビデオ通話を使う場合にしか機能せず、実際の面接で多用される Google Meet や Zoom を使うと AI 評価を利用できない状態でした。外部ツールを使う顧客への価値提供ができておらず、機能の普及を妨げていました。

この課題に対し、2 つの観点からソリューションを実装しました。まず、インタビューと Google カレンダーのイベントを紐づけることで、面接開始予定時刻に会議 Bot をカレンダーイベント内の会議 URL（Google Meet / Zoom）へ自動参加させ、会話をリアルタイムに文字起こしして AI 評価を可能にしました。Google Calendar API との連携と Google OAuth 認証をメインで担当しました。
また、発行時に面接官のカレンダーからイベントを選択することで Google カレンダーと連携し、面接官・候補者がカレンダー経由で会議 URL とインタビューリンクを受け取れる導線を構築しました。

**面接 Bot の参加タイミング改善と障害対応の強化**

リリース後、Bot が面接開始予定時刻に自動参加する仕様のため、技術面談が始まる前の自己紹介や雑談がそのまま AI 評価の対象に含まれてしまい、スキル評価の精度が低下する課題がありました。また、カレンダー予約の同期処理にバグがあり原因特定が困難な状態で、Bot の参加失敗時にリトライする手段もなく AI 評価が実施されないケースも発生していました。

バグ修正にコストをかけるより仕組みを刷新する判断をし、インタビューの「開始ボタン」をトリガーに Bot を参加させる方式に変更しました。これにより事前予約の同期処理を廃止してバグを根本的に解消し、同期処理のために存在していたリソースも削除することができました。あわせて面接官が任意のタイミングで Bot を参加させられるようになり、技術面談の内容のみを AI 評価の対象にできるようになりました。Bot 招待用の RPC も追加し、参加失敗時には直接 RPC を実行して再招待できる障害対応の仕組みも実装しました。

**生成 AI 共存型の新実践形式開発**

生成 AI の台頭によりアルゴリズム問題が容易にクラックされるようになり、従来のオンラインコーディングテストが提供してきた「効率的なスキル評価」というコアバリューが損なわれつつある課題がありました。一方でライブコーディングは生成 AI への耐性はあるものの企業側の工数が高く、スケールするソリューションにはなりません。

この課題を解決するため、生成 AI の使用を前提とした「生成 AI 共存型」の新しい試験形式を提案・実装しました。候補者は普段の開発環境で GitHub を通じて課題を受け取り、生成 AI の利用も含めて完全非同期で開発します。生成 AI で解けることを前提とした作問・評価を行うことで、生成 AI への耐性を気にすることなく候補者の実務スキルを測定できる仕組みを実現しました。

UI 開発では、課題の提出画面・提出コードの表示と差分表示・候補者共有レポートを担当しました。Monaco Editor を活用して初期コードと提出コードの差分を視覚的に表示し、レビュアーがコードの変更点を一目で把握できる UI を構築しました。

**一斉受験の負荷対策**

1〜2 時間の間に 1,000 名規模の同時アクセスが発生する一斉受験に向けて、システムが負荷に耐えられるか事前に検証する必要がありました。Biz サイドのメンバーと出題問題数・想定アクセス数を事前に認識合わせした上で、Tech Lead と協力して ghz（gRPC）と k6（HTTP）を使用した各 API への負荷検証を実施。無駄な SQL や処理を特定して最適化し、負荷を元の約 3/4 に削減しました。最適化後の再計測をもとに、Pod 数の増強・DB スペックアップ・オートスケールの Max 調整・CPU / メモリリソースの調整を行い、本番環境を整備しました。

初回開催当日に DB へのアクセスエラーが発生しましたが、原因を速やかに調査してチームに共有し対応。2 日目以降は問題なく全日程を完遂しました。

#### 事業横断の技術改善

事業開発と並行して、プラットフォーム全体の技術的負債の解消と安定性向上にも取り組みました。ボトルネッククエリの特定と最適化による大幅な高速化や、分散したマイクロサービスの統合による運用効率改善など、スケーラビリティと保守性の向上に貢献しています。

**Query の最適化**

API レスポンスが約 28 秒かかるパフォーマンス問題を調査しました。Cloud SQL のクエリインサイトとアプリケーション側のレイテンシ計測からボトルネックとなるクエリを特定し、EXPLAIN ANALYZE による実行計画の分析を実施。再帰 CTE 内で全カラムを取得していたことが、プランナーの非効率な実行計画選択と temp I/O の発生を招いていることを突き止めました。SELECT 対象を必要最小限のカラムに絞り、JOIN を明示化することで、実行時間を 20.6 倍高速化（3,053ms → 148ms）しました。

**マイクロサービスの統合**

マイクロサービスが細かく分散していたため、サービス間通信のオーバーヘッドと運用コストが増大していました。サービス境界を見直してマイクロサービスを統合し、通信オーバーヘッドの削減と運用負荷の低減を実現しました。

##### プロジェクト詳細

カスタマイズ全銀ファイルのデータ構造に合わせた独自のデータモデルを ERP 上に定義しました。カスタムレコードは 3 種類設計し、親ファイルデータ・全明細データ・突合に必要なデータのみを保存するレコードをそれぞれ作成。正規表現でマッチング条件を変更できる設定データも用意し、柔軟なルール変更を可能にしました。

突合の計算処理は独立した API として実装することで、ERP 標準機能を阻害しない設計を維持しつつ、将来的なマッチング精度の向上にも対応できる拡張性を確保しました。同時期に別の開発プロジェクトも並行していたため、実装範囲とデータモデルの影響範囲を事前に整理し、互いに干渉しない設計を徹底しました。

マッチング処理を含む細かなロジックは TDD で開発を進め、すべてテストコードを作成しました。実装に先立ちテストケースを整理し、コンサルタントおよびお客様と認識を合わせてから開発に着手することで、仕様の齟齬を早期に防ぎ、品質を担保した状態でリリースまで完遂しました。

##### プロジェクト詳細

**バリデーション不備による出荷遅延の緊急対応**

ERP と EC のデータ連携におけるバリデーション機能が正常に動作しておらず、在庫がない商品の注文や不正な住所での注文が通ってしまう状態でした。その結果、出荷処理が止まり納期遅延が発生していました。実際に登録されていた不正データを分析してバリデーション要件を整理し、問題発覚から数日以内に暫定対応を完了させました。

一次開発のコード品質が低くスパゲッティコード化していたため根本対応は困難でした。発生しているバグを一覧化してお客様に共有し、対応優先度の判断をお客様に委ねることで、限られたリソースを追加機能開発に集中できる体制を整えました。

**ERP-EC データ連携 API の刷新**

データ取得 API 内の SQL が重く、不要なデータまで取得していたためパフォーマンスが低下していました。TDD をベースに API のレスポンスを整理し、必要なデータのみを返す新しい API を作成して差し替えることで、連携処理の品質と速度を改善しました。

**追加機能開発とバグの段階的解消**

追加機能の開発においては、既存バグが残る箇所に近い実装が必要なケースがありました。その際に入力データの管理処理を一新し、バグが再発しない構造に整理しながら開発を進めました。お客様・コンサルタントと事前にテストケースで認識を合わせてから着手する TDD のアプローチを採用し、仕様齟齬を防いだ状態で機能をリリースしました。

出荷日指定の際に ERP のカレンダー情報を毎回取得していたため、操作のたびに不要な負荷が発生していました。初回アクセス時に必要な日付範囲のデータをまとめて取得してキャッシュする方式に変更し、繰り返しの取得処理を削減して負荷を軽減しました。

また、入力データや ID がブラウザのストレージに分散管理されており、キーの命名規則も存在しなかったため、どこで何が更新されるか追跡が困難な状態でした。キーの命名規則を定めて状態管理を整備した上で、ストレージへのアクセス処理をすべて単一の関数に集約しました。その関数を経由しなければストレージを更新できない設計に変更することで、不正な値の登録を防止し、データ管理の透明性と保守性を確保しました。
