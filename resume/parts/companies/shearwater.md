### Shearwater Japan 株式会社 (2023/11 ~ 2025/03)

ERP 導入・運用支援を専門とする企業にて、複数のお客様先プロジェクトに従事しました。業務要件のヒアリングから設計・開発・テストまでを一貫して担当し、ERP 標準機能では対応できない業務課題に対してカスタマイズ開発で解決策を提供しました。

入金消し込みのようなゼロからの設計・実装を含む複雑な開発案件に加え、ERP 稼働後のテクニカルサポートチームではメンバーマネジメントも担当。業務プロセスの整備・1on1 によるスキル支援・業務効率化スクリプトの開発を通じて、チームの対応品質と生産性の向上に貢献しました。

#### 導入後のテクニカルサポート （2024/06 ~ 2025/03）

ERP 稼働後のサポートチームにおいて、メンバーマネジメントを主軸に担当しました。問い合わせ対応・保守運用の品質向上に向けて、業務プロセスの整備・メンバー育成・作業効率化を推進しました。

##### プロジェクト規模

- サポートメンバー: 5 人 (自分含む)

##### プロジェクト詳細

**サポート業務プロセスの整理**

サポート対応の手順が属人化しており、担当者によって対応品質にばらつきがありました。業務プロセスを整理・標準化することで、チーム全体が一定品質で対応できる体制を構築しました。

**メンバーとの 1on1 によるスキル獲得支援**

メンバーごとに業務知識やシステム理解度にばらつきがあり、自律的な問い合わせ対応が難しい状況でした。定期的な 1on1 を実施してスキルギャップを把握し、各メンバーの習熟度に応じてフォローすることで、チーム全体の対応力を底上げしました。

**業務効率化のための Script 開発**

手作業が多く残っており、対応工数の削減が課題でした。問い合わせの通知機能をはじめ、サポート業務上の繰り返し作業を自動化する小規模なスクリプトを開発し、チームの作業効率を向上させました。

#### 入金消し込みカスタマイズ開発 (2024/10 ~ 2024/12)

ERP 標準機能では全銀ファイルを取り込んで債券データと突合・消し込みを行う機能を持たず、加えて本プロジェクトではカスタマイズされた全銀ファイルを使用していたため、標準機能では要件を満たせない状況でした。設計・実装・テスト設計をほぼ一人で担当し、要件を満たす仕組みをゼロから構築しました。

##### プロジェクト規模

- メンバー: 4 人
- コンサルタント: 1 人
- 開発: 1 人 (自分)
- テスター: 2 人

##### 使用技術

- JavaScript
- Oracle SQL
- GitHub

##### 役割

- 設計
- 実装
- テスト設計

##### プロジェクト詳細

カスタマイズ全銀ファイルのデータ構造に合わせた独自のデータモデルを ERP 上に定義しました。カスタムレコードは 3 種類設計し、親ファイルデータ・全明細データ・突合に必要なデータのみを保存するレコードをそれぞれ作成。正規表現でマッチング条件を変更できる設定データも用意し、柔軟なルール変更を可能にしました。

突合の計算処理は独立した API として実装することで、ERP 標準機能を阻害しない設計を維持しつつ、将来的なマッチング精度の向上にも対応できる拡張性を確保しました。同時期に別の開発プロジェクトも並行していたため、実装範囲とデータモデルの影響範囲を事前に整理し、互いに干渉しない設計を徹底しました。

マッチング処理を含む細かなロジックは TDD で開発を進め、すべてテストコードを作成しました。実装に先立ちテストケースを整理し、コンサルタントおよびお客様と認識を合わせてから開発に着手することで、仕様の齟齬を早期に防ぎ、品質を担保した状態でリリースまで完遂しました。

#### 服飾品メーカー ERP/EC 導入 (2023/10 ~ 2024/06)

途中から参画し、別会社が一次開発した EC サイトの引き継ぎと、ERP-EC 間のカスタマイズ開発・設計・保守を担当しました。一次開発のコードは低品質でバグも頻発しており、ERP と EC のデータ連携におけるバリデーション不備が出荷遅延を引き起こしていました。緊急の暫定対応から根本的な設計改善まで幅広く対応しました。

データ連携 API の刷新やブラウザストレージの状態管理整備など、コードの保守性・信頼性を高める改善を推進。追加機能開発においては TDD を採用し、お客様・コンサルタントとテストケースで認識を合わせてから着手することで、品質を担保した開発を継続しました。

##### プロジェクト規模

- メンバー: 3 ~ 4 人
- プロジェクトマネージャ: 1 人
- コンサルタント: 1 人
- 開発: 1 人 (自分含む)
- テスター: 1 人

##### 役割

- EC カスタマイズ開発/設計
- ERP システムとの連携開発
- 保守

##### 使用技術

- JavaScript
- Oracle SQL
- Sass

##### プロジェクト詳細

**バリデーション不備による出荷遅延の緊急対応**

ERP-EC データ連携のバリデーション機能は正常に動作しておらず、在庫のない商品や不正な住所での注文が通ってしまう状態でした。その結果、出荷処理が止まり、納期遅延を招いていました。実際に登録されていた不正データを分析してバリデーション要件を整理し、問題発覚から数日以内に暫定対応を完了させました。

一次開発のコード品質が低くスパゲッティコード化していたため根本対応は困難でした。発生しているバグを一覧化してお客様に共有し、対応優先度の判断をお客様に委ねることで、限られたリソースを追加機能開発に集中できる体制を整えました。

**ERP-EC データ連携 API の刷新**

データ取得 API 内の SQL が重く、不要なデータまで取得していたためパフォーマンスが低下していました。TDD をベースに API のレスポンスを整理し、必要なデータのみを返す新しい API を作成して差し替えることで、連携処理の品質と速度を改善しました。

**追加機能開発とバグの段階的解消**

追加機能の開発では、既存バグを含む箇所への実装を要するケースもありました。その際に入力データの管理処理を一新し、バグが再発しない構造に整理しながら開発を進めました。お客様・コンサルタントと事前にテストケースで認識を合わせてから着手する TDD のアプローチを採用し、仕様齟齬を防いだ状態で機能をリリースしました。

出荷日指定の際に ERP のカレンダー情報を毎回取得していたため、操作のたびに不要な負荷が発生していました。初回アクセス時に必要な日付範囲のデータをまとめて取得してキャッシュする方式に変更し、繰り返しの取得処理を削減して負荷を軽減しました。

また、入力データや ID がブラウザのストレージに分散管理されており、キーの命名規則も存在しなかったため、どこで何が更新されるか追跡が困難な状態でした。キーの命名規則を定めて状態管理を整備した上で、ストレージへのアクセス処理をすべて単一の関数に集約しました。その関数を経由しなければストレージを更新できない設計に変更することで、不正な値の登録を防止し、データ管理の透明性と保守性を確保しました。
